# 記事移行スキル（本番 → ローカルMarkdown）

本番サイト（hpkenkyu.mixh.jp）の公開記事HTMLをスクレイピングし、このプロジェクト独自のMarkdownパーサーが解釈できる形式に変換するスキル。

**ゴール**: 本番の見出し構造・段落順序・テーブル・画像配置を忠実に再現しつつ、利用可能な装飾パーツで適切にマークアップすること。

---

## Step 1: 自動変換スクリプトの実行

同梱のPythonスクリプトで一括変換する。BeautifulSoup4ベースのDOMパーサーで、テーブル内の画像・リンクも自動抽出する。

```bash
# stdoutに出力（確認用）
python3 .claude/skills/migrate-article/scripts/convert_html_to_md.py <URL>

# ファイルに直接出力
python3 .claude/skills/migrate-article/scripts/convert_html_to_md.py <URL> meikan/content/articles/<slug>.md
```

※ `beautifulsoup4` と `requests` が必要。未インストールなら `pip install beautifulsoup4 requests` を実行。

スクリプトが自動処理する内容:
- entry-content divの抽出とHTML→Markdown変換
- テーブル内の`<img>`→`!img[URL]`、`<a>`→`[Name](URL)`の個別パース
- 旧サイト内部リンクの自動差し替え（マッピング表 + 既存記事slug照合）
- SANGO装飾ボックスの対応パーツへの変換
- CTAボタンの`[btn テキスト](URL)`変換
- 目次・SNSボタン・関連記事カード等の自動除外
- フロントマター自動生成（noindex: true付き）

## Step 2: スクリプト出力の確認と手動補完

スクリプトの出力をそのまま使えることが多いが、以下の点は手動で確認・補完が必要。

### 基本変換ルール

| HTML | Markdown |
|------|----------|
| `<h2>` | `## 見出し` |
| `<h3>` | `### 見出し` |
| `<p>` | 段落テキスト（空行で区切る） |
| `<ul><li>` | `- リスト項目` |
| `<ol><li>` | `1. リスト項目` |
| `<table>` | Markdownテーブル記法 |
| `<figure><img>` | `![alt](src)` |
| `<blockquote>` | `> 引用テキスト` |
| `<strong>` / `<b>` | `**太字**` |
| `<a href="url">text</a>` | `[text](url)` |

### 画像URL

画像のsrcは以下の優先順で取得:
1. `data-src` 属性（遅延読み込み用）
2. `src` 属性

WordPress固有のサイズサフィックス（`-1024x685` 等）はそのまま維持。

### WordPressの画像テーブル

本番では一部の比較表がWordPressの画像（スクリーンショット）として埋め込まれている場合がある。これらは本文のコンテキストから内容を推測し、Markdownテーブルに置き換える。画像テーブルは以下のパターンで判別できる:
- 表タイトルのテキスト（例: 「アダルト動画配信サービス6社比較」）の直後に画像がある
- 前後の本文で表の内容に言及している

置き換え不可能な場合はそのまま画像として残す。

### テーブル内のロゴ画像とリンク

本番のHTMLテーブルにはサービスロゴ画像やアフィリエイトリンクが含まれている場合が多い。これらを忠実に再現する。

**ロゴ画像**: テーブルセル内の`<img>`から画像URLを抽出し、`!img[URL]`構文でサービス名の前に配置する。

**リンク**: テーブルセル内の`<a>`からリンク先URLを抽出し、サービス名を`[Name](URL)`でリンク化する。
- アフィリエイトリンク（`al.dmm.co.jp`等の外部URL）はそのまま維持
- 旧サイトの内部リンク（`/sokmill-link/`等）は対応する公式サイトURLに差し替え

**変換例**:
```markdown
| !img[https://example.com/logo.webp] [サービス名](https://example.com/) | 42万本以上 |
```

テーブルセル内の画像・リンクを抽出するには、HTMLテーブルの各`<td>`の中身を個別に解析する必要がある。`re.findall`で`<img>`と`<a>`を別々に取得してからMarkdownに組み立てること。

### CTAボタンの変換

本番でボタン形式のリンク（`wp-block-button`クラスや`btn`クラスを持つ`<a>`要素）がある場合、`[btn テキスト](URL)`構文に変換する。旧サイト内部リンクの場合は対応する外部URLに差し替える。

### 除外するHTML要素

- 目次（TOC）: 「目次」テキスト + リスト形式のアンカーリンク群
- SNSシェアボタン: Twitter/Facebook/はてブ/LINE のシェアリンク
- カテゴリリンク: `CATEGORY :` 以降
- 関連記事カード: 末尾のリンクカード群
- 「前の記事」「次の記事」ナビゲーション
- WordPress固有の装飾クラス（`.is-style-xxx`, `.wp-block-xxx`）
- 広告・PR表記（ただし記事冒頭の「PR」テキストは削除）

### 旧サイト内部リンクの処理

`https://hpkenkyu.mixh.jp/` 配下へのリンクは基本的に削除する。ただし:
- 引用元リンク（Yahoo!知恵袋等の外部サイト）は維持
- アフィリエイトリンク等の外部リンクは維持

## Step 3: 装飾パーツへの変換

本番のHTML要素を、このプロジェクトで利用可能な以下のMarkdown装飾パーツに変換する。

### 利用可能な装飾パーツ一覧

#### :::proscons — メリット・デメリット比較表
本番でメリット・デメリットが並べて表示されている場合に使用。

```markdown
:::proscons[タイトル]
pros[メリットのラベル]:
- 項目1
- 項目2
cons[デメリットのラベル]:
- 項目1
- ==[red]強調したい項目==
:::
```

#### :::alert — 警告・注意書き
本番で専用の装飾ボックスとして囲われている警告・メンテナンス情報に使用。CSSが⚠️アイコンをタイトルの前に自動付与するため、タイトルに⚠️を含めないこと。

プレーンテキストで書かれた警告文（装飾ボックスに囲われていないもの）は`:::alert`で囲わず、そのまま段落として出力する。`==[red]テキスト==`で強調するだけで十分。

```markdown
:::alert[タイトル]
警告本文テキスト
:::
```

#### :::box — 汎用ボックス
本番で枠囲みされた情報ブロックがある場合に使用。

```markdown
:::box[タイトル]
- リスト項目
- リスト項目
:::
```

#### :::memo — メモボックス
補足情報や豆知識的な内容に使用。

```markdown
:::memo[タイトル]
メモ内容
:::
```

#### :::faq — FAQアコーディオン
Q&A形式の内容に使用。

```markdown
:::faq[質問テキスト]
回答テキスト
:::
```

#### :::say — 吹き出し（av博士コメント）
筆者の個人的な感想・コメントがある場合に使用。

```markdown
:::say
コメントテキスト
:::
```

#### :::chat — チャット会話
会話形式のコンテンツに使用。

```markdown
:::chat
話者A: 発言内容
話者B: 発言内容
:::
```

#### インライン装飾

| 構文 | 用途 |
|------|------|
| `==テキスト==` | 黄色ハイライトマーカー |
| `==[red]テキスト==` | 赤ハイライトマーカー |
| `**テキスト**` | 太字 |
| `[btn テキスト](URL)` | CTAボタンリンク |
| `!img[URL]` | インライン画像 |

### WordPress装飾クラスとの対応表

| WordPress側の装飾 | 変換先パーツ |
|------------------|------------|
| `.marker`/`.marker-under`（蛍光ペン黄） | `==テキスト==` |
| `.marker` + red系クラス（蛍光ペン赤） | `==[red]テキスト==` |
| SANGO系の装飾ボックス | `:::box` / `:::memo` / `:::alert` |
| 吹き出しブロック | `:::say` |
| メリデメ比較テーブル | `:::proscons` |

## Step 4: フロントマターの生成

以下のフロントマターを生成する:

```yaml
---
title: <h1タグまたはmeta titleから取得（サイト名部分は除去）>
slug: <URLパスから取得（例: /sokmil/ → sokmil）>
description: <meta descriptionから取得>
category: <ユーザーに確認。不明なら空欄>
published_at: <今日の日付（YYYY-MM-DD）>
updated_at: <今日の日付（YYYY-MM-DD）>
noindex: true
---
```

- `noindex: true` はデフォルトで付与（移行直後はインデックスさせない）
- `category` はユーザーに確認する。既存カテゴリが不明な場合は記事内容から推測して提案

## Step 5: ファイル出力

完成したMarkdownを以下のパスに保存:

```
meikan/content/articles/{slug}.md
```

## Step 6: 検証

出力したファイルの品質を確認:

1. ローカルサーバーが起動していれば `http://localhost:8000/article/{slug}/` にアクセスしてHTML出力を確認
2. 以下の項目をチェック:
   - 見出し構造が本番と一致しているか
   - テーブルが正しくレンダリングされているか
   - **テーブル内のロゴ画像が表示されているか**（本番HTMLの`<td>`内`<img>`と突き合わせる）
   - **テーブル内のサービス名にリンクが付いているか**（本番HTMLの`<td>`内`<a>`と突き合わせる）
   - **CTAボタンが全て変換されているか**（本番HTMLのbtn/button系`<a>`と突き合わせる）
   - 画像が表示されるか（URLの有効性）
   - 装飾パーツが正しく変換されているか
   - 旧サイトへの内部リンクが残っていないか

---

## 注意事項

- **順序の忠実性が最優先**: 自然言語で「こう変えて」と指示するのではなく、HTMLの出現順序をそのまま維持する。これが最も重要なルール。
- **過度な編集をしない**: 本番のテキストを勝手にリライトしない。文言はそのまま移行する。
- **不明な装飾は素通し**: 対応するパーツがない装飾は、プレーンテキストとして出力する。無理に変換しない。
- **画像URLはそのまま**: WordPressのアップロードURLをそのまま使用する。ローカルへのダウンロードは行わない。
